# KairoLogic Dashboard — Design Principles

These principles govern all workflow and dashboard UI decisions.
Follow them when designing new workflows, modifying existing ones, or adding features.

---

## 1. Fewest Clicks to Complete

Every workflow should require the absolute minimum number of user actions.
- No redundant buttons, checkboxes, or confirmation steps that repeat the same intent.
- If a user clicks "Download PDF", that IS the completion of the download step — don't add a separate checkbox.
- If a user clicks "I've submitted this", that IS both the submission acknowledgment and the task completion — no separate "Mark as submitted" button.
- Merge steps that always happen together into a single step.

## 2. No Redundant UI Elements

- Never show two buttons/controls that do the same thing.
- Never show a checkbox AND a button for the same action.
- If an action is automatic (e.g., monitoring), show status — don't ask the user to do anything.
- Each screen should have exactly one primary action.

## 3. Provider Roster = Completed Providers Only

- Providers are NOT added to the roster until their onboarding workflow is fully complete.
- During onboarding, the provider exists only as a workflow.
- The roster shows active, departing, and departed providers — never "onboarding".
- Cancelling an onboarding workflow means the provider never joins the roster.

## 4. Every Workflow Has a Cancel Option

- All active workflows show a "Cancel workflow" button with a confirmation step.
- Cancelling skips all pending tasks, marks the workflow cancelled, and dismisses related alerts.
- Cancelled workflows remain visible in the audit trail for compliance.

## 5. Documents Are Auto-Saved

- Any artifact generated during a workflow (PDF forms, correction packets, checklists) must be saved to `workflow_artifacts` automatically.
- Generated documents appear in the Documents page with metadata: provider name, workflow type, generation date, generated by.
- Users should never need to manually save or re-download a document — it's always in Documents.

## 6. Full Audit Trail for Compliance

- Every workflow action records: what happened, who did it, when, and relevant details.
- "Who" = the authenticated user (from Supabase auth), not just "user" vs "system".
- Task completions, approvals, downloads, cancellations, and status changes all log to `workflow_events`.
- The audit trail is immutable — events are never deleted, even on workflow cancellation.
- Fields: `event_type`, `actor_type` (user/system), `actor_id` (user UUID), `actor_email`, `title`, `details` (JSON).

## 7. Workflow Step Design Pattern

Each workflow follows this pattern:
1. **Action steps** — require one clear user action each (review, approve, download, submit)
2. **Automated steps** — show status only, no user action needed (monitoring, auto-confirm)
3. Steps activate sequentially; automated steps can run in parallel with the final action step.
4. When all steps complete, the workflow auto-resolves and triggers any downstream effects (e.g., add to roster).

## 8. Inline Over Navigate

- Prefer modals and slide-over panels over page navigation for single-entity actions.
- "Add provider" = modal, not a new page.
- "View workflow detail" = slide-over panel, not a new page.
- Only use full page navigation for list views (Workflows, Roster, Alerts, Documents).

## 9. Status Badges Are Consistent

- Use the same status colors and labels everywhere: `statusColors`, `statusLabels` from `design-tokens.ts`.
- Workflow types use gold pills. Workflow statuses use navy pills. Roster statuses use their own color map.
- Never invent ad-hoc status colors.

## 10. Progressive Disclosure

- Show summary first, details on click/expand.
- Workflow cards show: type, provider, summary, status, due date.
- Workflow detail panel shows: full task checklist, source comparison, timeline, documents.
- Don't overwhelm the practice manager with everything at once.
